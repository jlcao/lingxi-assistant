# 全链路异步改造说明文档

## 改造概述

本次改造将灵犀智能助手从同步架构全面升级为异步架构，彻底解决 WebSocket 消息阻塞问题。

## 架构对比

### 改造前（同步架构）

```
WebSocket 请求 → FastAPI 事件循环 → 同步引擎（阻塞） → LLM API（阻塞） → 技能调用（阻塞）
                      ↓
              ❌ 事件循环被阻塞
              ❌ 其他 WebSocket 连接等待
              ❌ 并发性能差
```

### 改造后（异步架构）

```
WebSocket 请求 → FastAPI 事件循环 → 异步引擎（非阻塞） → 异步 LLM API（非阻塞） → 异步技能调用（非阻塞）
                      ↓
              ✅ 事件循环自由
              ✅ 所有连接并行处理
              ✅ 高并发性能
```

## 核心改动

### 1. 异步 LLM 客户端 (`lingxi/core/async_llm_client.py`)

**关键特性**：
- 使用 `httpx.AsyncClient` 进行异步 HTTP 请求
- 支持异步流式响应 (`async for chunk in stream_chat(...)`)
- 自动重试机制
- Token 使用统计

**示例代码**：
```python
from lingxi.core.async_llm_client import AsyncLLMClient

async_llm = AsyncLLMClient(config)

# 流式聊天
async for chunk in async_llm.stream_chat(messages, task_level="complex"):
    print(chunk)

# 非流式聊天
response = await async_llm.chat(messages)
```

### 2. 异步技能调用器 (`lingxi/core/skill_caller.py`)

**关键特性**：
- 保留同步方法（向后兼容）
- 新增异步方法 `call_async()`
- 使用线程池执行同步技能（过渡方案）

**示例代码**：
```python
# 异步调用技能
result = await skill_caller.call_async("search", {"query": "天气"})
```

### 3. 异步引擎核心 (`lingxi/core/engine/async_react_core.py`)

**关键特性**：
- 继承自 `AsyncReActCore`
- 所有方法都是 `async def`
- 使用异步生成器 (`async def ...() -> AsyncGenerator[...]`)
- 异步 LLM 调用
- 异步事件发布

**关键方法**：
- `async def _execute_task_stream(context)` - 异步流式执行任务
- `async def _process_llm_response(messages)` - 异步处理 LLM 响应
- `async def process(context)` - 异步处理用户输入

### 4. 异步 Plan+ReAct 引擎 (`lingxi/core/engine/async_plan_react.py`)

**关键特性**：
- 继承自 `AsyncReActCore`
- 支持复杂任务规划
- 异步计划执行
- 检查点保存/恢复（异步）

### 5. 异步助手类 (`lingxi/core/async_main.py`)

**关键特性**：
- `AsyncLingxiAssistant` 类
- 所有方法都是异步的
- 使用异步引擎

**示例代码**：
```python
from lingxi.core.async_main import AsyncLingxiAssistant

assistant = AsyncLingxiAssistant(config)

# 异步处理输入
response = await assistant.process_input("你好", session_id="test")

# 异步流式处理
async for chunk in assistant.stream_process_input("写一首诗"):
    print(chunk)
```

### 6. 异步 WebSocket 层 (`lingxi/web/websocket.py`)

**关键改动**：
- 使用 `AsyncLingxiAssistant`
- 直接 `async for` 遍历异步生成器
- 无需线程池/队列

**关键代码**：
```python
async def _send_stream_response(self, connection, message, session_id):
    # 直接调用异步助手
    response_generator = await self.assistant.stream_process_input(message, session_id)
    
    # 直接遍历异步生成器并发送
    async for chunk in response_generator:
        await connection.send_json(chunk)
```

### 7. 异步 FastAPI 服务器 (`lingxi/web/fastapi_server.py`)

**关键改动**：
- 启动时创建 `AsyncLingxiAssistant`
- WebSocket 端点使用异步管理器

## 使用指南

### 启动异步 WebSocket 服务器

```bash
python start_web_server.py --reload
```

服务器会自动使用 `AsyncLingxiAssistant`。

### 在代码中使用异步助手

```python
import asyncio
from lingxi.core.async_main import AsyncLingxiAssistant

async def main():
    assistant = AsyncLingxiAssistant("config.yaml")
    
    # 非流式
    response = await assistant.process_input("你好")
    print(response)
    
    # 流式
    async for chunk in assistant.stream_process_input("写一首诗"):
        print(chunk["content"], end="", flush=True)

asyncio.run(main())
```

### 异步引擎直接使用

```python
from lingxi.core.engine.async_plan_react import AsyncPlanReActEngine
from lingxi.core.context import TaskContext

engine = AsyncPlanReActEngine(config, skill_caller, session_manager)

context = TaskContext(
    user_input="查询北京天气",
    task_info={"level": "simple"},
    session_id="test",
    session_history=[],
    stream=True
)

async for chunk in engine.process(context):
    print(chunk)
```

## 常见问题

### Q1: 为什么不全部重写为异步？

**A**: 为了保持向后兼容性和降低风险。我们采用了渐进式改造策略：
- 新增异步版本，保留同步版本
- WebSocket 层使用异步版本
- 其他模块可以逐步迁移

### Q2: 异步架构有什么缺点？

**A**: 
- 代码复杂度略高（需要 `async/await`）
- 调试稍复杂（异步堆栈）
- 需要异步思维的转变

### Q3: 性能提升明显吗？

**A**: 
- **IO 密集型任务**：提升显著（10-100x 并发）
- **CPU 密集型任务**：提升有限（需要多进程）
- **WebSocket 场景**：提升巨大（不再阻塞）

### Q4: 还需要线程池吗？

**A**: 
- **技能调用**：暂时需要（过渡方案）
- **LLM 调用**：不需要（已异步化）
- **引擎执行**：不需要（已异步化）
